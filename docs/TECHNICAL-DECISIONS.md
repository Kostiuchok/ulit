## 📦 Архітектура

### Чому Turborepo monorepo?

`web`, `api`, `worker` — три окремі Docker-образи з різними базовими образами (Alpine vs Ubuntu). Monorepo дозволяє:
- Спільні TypeScript типи через `packages/shared-types` без публікації в npm
- Єдиний `pnpm-lock.yaml` — детермінована збірка
- Turbo кешує build/lint/test — швидше в CI

### Чому три окремі застосунки (web / api / worker)?

- **web** (Next.js): SSR, SEO, клієнтська частина — потребує Node 20 Alpine
- **api** (Fastify): REST API, бізнес-логіка — легкий Alpine образ, ~200MB
- **worker** (BullMQ): обробка файлів через LibreOffice/Pandoc/Ghostscript/Calibre — потребує Ubuntu 22.04, ~1.5GB. Не можна поєднати з api бо системні залежності забруднять образ.

### Чому Fastify, а не Express або Hono?

- Fastify швидший за Express (benchmark: 2–3x)
- Вбудована схема валідації через JSON Schema
- Plugin система з version checking (уникає несумісних плагінів)
- TypeScript support з коробки
- Hono розглядався але Fastify має кращу екосистему для Prisma/BullMQ

### Чому PostgreSQL, а не MongoDB?

Дані платформи — реляційні: User → Book → Order → OrderItem → Royalty. PostgreSQL дає:
- Foreign keys та транзакції для фінансових операцій
- `tsvector` для fulltext пошуку (без ElasticSearch)
- JSONB якщо потрібна гнучкість
- Prisma ORM підтримує PostgreSQL краще за MongoDB

### Чому MinIO, а не S3?

- Self-hosted на тому ж VPS — zero egress cost
- S3-сумісний API — при масштабуванні легко перейти на AWS S3 або Cloudflare R2
- Файли авторів (DOCX, EPUB, PDF) — великі, постійний доступ через signed URLs

## 🔐 Авторизація

### NextAuth.js v5 + JWT strategy

- **Чому NextAuth**: готова інтеграція з Next.js, підтримує Google OAuth без зайвого коду
- **JWT (не database sessions)**: авторизація без зайвих запитів до БД на кожен request
- **Env змінні NextAuth v5**: `AUTH_SECRET` (не `NEXTAUTH_SECRET`), `AUTH_URL` (не `NEXTAUTH_URL`)
- **Google OAuth**: додатковий канал реєстрації, популярний серед авторів
- **Caddy routing**: `/api/auth/*` → `knyha-web:3000` (Next.js обробляє), `/api/*` → `knyha-api:3001` (Fastify)

### API JWT (Fastify)

Fastify використовує окремий JWT для авторизації API запитів (middleware). Секрет зберігається в `NEXTAUTH_SECRET` env змінній API. Це НЕ той самий секрет що NextAuth використовує для своїх сесій — вони можуть збігатись або різнитись.

## 📚 Обробка книг

### Формати та конвертація

| Формат | Інструмент | Призначення |
|---|---|---|
| PDF (screen) | LibreOffice headless | Онлайн перегляд, продаж |
| PDF/X-3 (print) | Ghostscript | Друкарня — CMYK, 300 DPI, bleed 3мм |
| EPUB 3 | Pandoc | Електронні читалки, D2D, Google |
| FB2 | Calibre | Популярний у СНД формат |
| MOBI/AZW3 | Calibre | Amazon Kindle |

### Чому Ubuntu 22.04 для worker (не Alpine)?

LibreOffice, Pandoc, Ghostscript і Calibre мають складні системні залежності (glibc, fonts, python). На Alpine вони або не збираються або потребують musl compatibility шарів. Ubuntu 22.04 — стабільна база, пакети в apt. Розмір образу ~1.5GB — прийнятно для worker.

### Черга BullMQ + Redis

- Конвертація може тривати 30–120 секунд — не можна блокувати HTTP request
- BullMQ дає retry логіку, delay, priority, dead letter queue
- Redis вже використовується для кешу — одна інфраструктура
- Кожен формат — окремий job (паралельна конвертація)

### Валідація print PDF

Якщо PDF/X-3 (print) < 10MB — повторити без стиснення. Маленький розмір = втрата якості неприйнятна для друкарні.

## 🌍 Дистрибуція — ручна vs автоматична

### Поточна модель (MVP): повністю ручна

Адмін отримує чергу книг → завантажує ZIP пакет (EPUB + обкладинка + metadata.csv) → вручну завантажує на кожен сервіс.

| Сервіс | Статус | Причина |
|---|---|---|
| **Amazon KDP** | Ручна | KDP Selling Partner API вимагає офіційної заявки та схвалення Amazon. Процес: подати заявку → чекати схвалення (тижні/місяці) → отримати API credentials |
| **Draft2Digital (D2D)** | Ручна | D2D Partner API вимагає Business Development угоду. Контакт: bizdev@draft2digital.com |
| **Google Play Books** | Ручна | Google Books Partner Center API — закрита програма, потребує заявки |
| **Власний магазин** | Автоматична | Одразу після публікації |

### Стратегія дистрибуції автора

**WIDE (Широке)**: книга потрапляє в чергу адміна → D2D + KDP + Google (вручну)  
**KDP_SELECT (Ексклюзив Amazon)**: тільки Amazon 90 днів. Система автоматично знімає з інших платформ. За 7 днів до закінчення — email автору.

### Майбутня автоматизація (Фаза 2)

- Після схвалення KDP SP API → автоматично завантажувати через API
- D2D планує відкрити API — слідкувати за їх блогом
- Google Books — при партнерстві Google надає доступ до Content API

## 💳 Платежі

### Вибір провайдерів

| Провайдер | Статус | Для кого |
|---|---|---|
| **LiqPay** | Реалізовано (MVP) | Українські картки (Приват24, Монобанк, Visa/MC) |
| **WayForPay** | Планується (Фаза 2) | Альтернатива LiqPay, популярна в Україні |
| **Stripe** | Планується (Фаза 2) | Міжнародні картки, Google Pay, Apple Pay |

### Чому LiqPay першим?

- Найпоширеніший в Україні — інтеграція з Приват24 і Монобанком
- Проста інтеграція через form + signature
- LiqPay Payback API — майбутні виплати авторам на картки Приват24

### Архітектура callback

Після оплати → LiqPay → `POST /api/payments/liqpay/callback` → verify signature → оновити Order.status = PAID → надіслати email з signed URLs (48 год).

## 💸 Система виплат авторам

### Бізнес-модель

- **Комісія платформи**: 25% від кожного продажу (змінюється через `PLATFORM_FEE_PERCENT` env)
- **Автор отримує**: 75% від ціни книги мінус комісія платіжного провайдера (~2.5%)
- **Мінімальний поріг виплати**: 500 UAH

### Hold period (витримка 7 днів)

Гроші блокуються на 7 днів після продажу перед виплатою. Причина: захист від chargebacks (покупець може скасувати оплату протягом 7-14 днів). Практика всіх marketplace (Etsy, Gumroad, Amazon).

### Виплата Phase 1 (ручна)

Автор запитує виплату → адмін бачить чергу → відправляє вручну через LiqPay або банківський переказ → позначає як виплачено.

### Виплата Phase 2 (автоматична)

- LiqPay Mass Payments B2P API → автоматичний переказ на картку Privat24
- WayForPay Settlement API → будь-яка українська картка
- IBAN (SWIFT/SEPA) → для ФОП з розрахунковим рахунком

### Податкова відповідність (Україна)

- При реєстрації автор вказує тип: **фізична особа** або **ФОП (Фізична особа-підприємець)**
- Зберігається **РНОКПП** (реєстраційний номер облікової картки платника податків)
- ФОП самостійно сплачує податки (єдиний податок)
- Фізособа — платформа має інформувати про необхідність декларування доходів
- CSV звіт для бухгалтерії: ПІБ, РНОКПП, сума, дата виплати

## 📜 Юридика

### Публічна оферта (договір з автором)

Автор акцептує договір при першій публікації. Зберігається:
- `contractAcceptedAt` — timestamp акцепту
- IP адреса — для доказової бази

Договір розміщено на `/author-agreement`. Зміна договору потребує нового акцепту.

### GDPR / Cookie

- Cookie banner при першому відвіданні
- Мінімальні дані: email, ім'я, IP (тільки для договору)
- Право на видалення акаунту — реалізувати в Фазі 2

### Авторські права

- Автор підтверджує що є власником авторських прав при публікації
- Платформа не несе відповідальності за порушення — зазначено в договорі

## 🏗️ Інфраструктура

### VPS vs хмара

Обрано **VPS (Hetzner/HostGator)** а не AWS/GCP:
- Фіксована вартість (~$20-40/місяць vs $200+ на AWS для аналогічного навантаження)
- MinIO self-hosted — без S3 costs
- Простота на MVP стадії

### Reverse proxy: Caddy

- Caddy автоматично отримує та оновлює SSL сертифікати (Let's Encrypt)
- Простіша конфігурація ніж Nginx
- **Важливо**: Caddy (`dddcore-caddy-1`) — окремий проект на сервері. НЕ модифікувати без потреби.
- Routing: `/api/auth/*` → Next.js `:3000`, `/api/*` → Fastify `:3001`, все інше → Next.js `:3000`

### Docker networks

- `caddy` (= `dddcore_default`): web + api → доступні з Caddy
- `internal`: api + worker + postgres + redis + minio → ізольовані від зовнішнього світу
- `monitoring`: prometheus + grafana + node-exporter

### Monitoring

Grafana + Prometheus для метрик VPS (CPU, RAM, disk). Доступ через SSH tunnel:
```bash
ssh -L 3003:localhost:3000 user@vps  # Grafana
ssh -L 9090:localhost:9090 user@vps  # Prometheus
```

## 🎨 UX рішення

### Wizard публікації (5 кроків)

1. Завантаження DOCX
2. Редактор обкладинки (Fabric.js canvas)
3. Метадані (назва, опис, жанр, ціна)
4. Стратегія дистрибуції (WIDE / KDP_SELECT)
5. Підтвердження + публікація

Wizard зберігає стан між кроками (Zustand). Автор може повернутись на будь-який крок.

### Preview уривку

Автор виділяє сторінки для безкоштовного preview. Покупець читає через `epubjs` reader з темами (день/ніч). При досягненні кінця preview — paywall.

### Cover Designer (Fabric.js)

Canvas-based редактор обкладинки. Підтримує:
- Завантаження власного зображення
- Безкоштовні фони через Unsplash API
- Текстові шари, шрифти, кольори
- 5+ готових шаблонів
- Експорт PNG 300 DPI
